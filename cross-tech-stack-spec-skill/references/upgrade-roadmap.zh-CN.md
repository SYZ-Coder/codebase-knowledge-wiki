# 跨技术栈扩展技能升级清单

本文记录在一个具有代表性的真实混合工作区完成验证之后，识别出的主要升级方向。

它重点回答三个问题：

- 下一步应该升级什么
- 这个能力应该放进主技能，还是继续留在扩展技能
- 团队落地使用时，什么顺序更合理

## 1. 当前成熟度判断

这个扩展技能现在已经可以用于：

- 移动端项目
- H5 / Web 前端项目
- Python 服务 / worker / AI 项目
- 同时包含 app、page、bridge、backend、task、消息链路的混合工作区

这次真实项目验证已经证明，它能够从代码中抽出以下证据：

- 原生路由
- App 到 H5 的跳转
- H5 到后端的 HTTP 调用
- H5 到 Native 的桥接调用
- IM SDK 接入
- FastAPI 路由
- SSE
- Kafka
- Redis
- 异步 HTTP 客户端

但是，它还没有达到“团队长期大规模稳定复用、几乎不用补规则”的程度。

## 2. 优先升级项

### 优先级 1：加强工作区分层

问题：

在大体量混合工作区里，如果一开始不先建立分析边界，直接全量扫描会产生大量噪音。

升级方向：

无论后续做什么分析，都优先支持一页“工作区分层页”，至少固定输出：

- workspace
- subsystem
- repository
- stack type
- entry files
- recommended analysis order

作用：

让后续所有文档都站在同一套边界之上，减少分析漂移。

建议归属：

- 主技能：吸收通用的 workspace classification 思想
- 扩展技能：保留混合栈专用的分层规则

### 优先级 2：给通信分析增加证据强度等级

问题：

现在已经能找到通信证据，但输出还不够容易区分“强事实”和“弱线索”。

升级方向：

每条通信结论都带上证据强度，例如：

- L1 直接声明
- L2 配置证据
- L3 线索级证据

示例：

- L1：`@POST("/api/...")`、`include_router(...)`、`import TIM from 'tim-js-sdk'`
- L2：`baseURL`、`kafka:` 配置、`redis:` 配置
- L3：命名、目录、适配器名等线索，但没有直接调用点

作用：

提高可信度，也更符合“严格按事实、不额外推理”的原则。

建议归属：

- 主技能：吸收通用的证据等级纪律
- 扩展技能：保留不同通信类型下的具体例子

### 优先级 3：内建噪音排除规则

问题：

混合仓库里常常包含训练代码、模型代码、生成物、第三方目录，这些内容很容易淹没业务主链路。

升级方向：

扩展技能明确优先扫描：

- main
- routers
- controllers
- api
- request
- services
- bridge
- im
- mq
- kafka
- task entrypoints

同时明确降权或延后扫描：

- vendor
- third_party
- model_zoo
- training 目录
- 生成产物目录
- 与链路分析无关的大量静态资源

作用：

让输出更接近“业务知识地图”，而不是“源码文件罗列”。

建议归属：

- 主技能：保留通用排除原则
- 扩展技能：维护混合栈专用排除列表

### 优先级 4：补一层前后端接口对照页

问题：

当前已经能分别抽取前端接口声明和后端路由声明，但还没有稳定输出“前端请求 -> 后端路由”的对照页。

升级方向：

增加可选输出：

- frontend caller
- request path
- request method
- backend route
- code locations
- 是否 direct / partial / unresolved

作用：

让接口链路更明朗，也方便跨团队联调和知识复用。

建议归属：

- 主技能：未来可以吸收这种接口对照输出模式
- 扩展技能：先实现首版，因为这个问题在混合栈项目里更突出

### 优先级 5：升级业务域页的跨技术栈映射

问题：

原来的 `domain -> service -> spec` 结构，在混合仓库里仍然有一点偏后端视角。

升级方向：

对混合栈场景，把域页扩成：

- domain
- entrypoints / surfaces
- systems / services / modules
- rules / specs

其中 entry surfaces 可以包括：

- app 页面
- H5 页面
- backend API
- Python task
- bridge entrypoints
- callback

作用：

让业务域页真正变成团队上下文页，而不是后端总结页。

建议归属：

- 主技能：继续保留稳定的域级结构
- 扩展技能：提供跨栈适配层

### 优先级 6：支持增量更新模式

问题：

当前流程更适合第一次从头梳理，不够适合长期维护。

升级方向：

增加类似 `update_codemap` 的增量模式，只更新变化区域，并固定输出：

- change scope
- newly confirmed evidence
- affected documents
- unresolved parts

作用：

如果未来要进入中央知识库长期维护，这是必需项。

建议归属：

- 主技能：吸收通用的增量更新纪律
- 扩展技能：补充混合栈场景下的更新示例与启发规则

## 3. 哪些适合进入主技能

以下改进足够通用，后续应该进入稳定主流程：

- 工作区分类作为标准第一步
- 证据强度纪律
- 增量更新纪律
- 文档影响范围记录

原因：

这些并不只属于移动端、H5 或 Python，它们本质上是在提升整个项目分析流程的可靠性。

## 4. 哪些继续留在扩展技能

以下能力更适合继续保留在扩展层：

- app / H5 / Python / mixed repo 的栈专用分类规则
- bridge 导向的通信分析
- app 到 H5、H5 到 Native 的链路模式
- IM / SSE / callback / task / mixed-entry 的通信示例
- 混合栈噪音排除规则
- 业务域页的跨技术栈适配

原因：

这些能力很有价值，但如果过早放进主技能，会污染目前已经稳定的后端导向主心智模型。

## 5. 建议升级顺序

一个比较实用的升级顺序是：

1. 先加强工作区分层
2. 再加入证据强度等级
3. 再加入噪音排除规则
4. 然后补前后端接口对照页
5. 再升级跨技术栈业务域页
6. 最后做增量更新模式

原因：

前三项先提升结果可靠性。
后两项提升结果可用性。
最后一项提升长期维护能力。

## 6. 对团队使用方式的建议

团队在使用这个扩展技能时，最稳的顺序是：

1. 先做工作区分类
2. 再做单仓或单子系统 codemap
3. 再做跨层 router-map
4. 最后再做业务域页

这一点对业务域梳理尤其重要。

在混合工作区里，只有当底层 codemap 和 router-map 已经稳定后，业务域页才会更可靠。

## 7. 结论

这个扩展技能不需要推倒重来。

它现在最需要补强的是：

- 更强的边界
- 更强的证据标注
- 更强的长期维护支持

这三点，决定了它只是“能分析一次混合项目”，还是“能成为团队中央知识库长期可信赖的一部分”。
