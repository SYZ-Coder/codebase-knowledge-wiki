# 跨服务业务域梳理

## 1. 目标

业务域页的价值，不是重复服务页，而是把零散服务知识提升成团队可复用的上下文层。

## 2. 三层关系

推荐结构：

- 域
- 服务
- 规范

### 2.1 域层

回答：

- 这个域是什么
- 当前哪些服务强事实承载这个域
- 哪些服务只是线索级关联
- 域内有哪些核心对象、能力、协作关系

推荐文件：

- `overview.md`
- `services.md`
- `rules.md`
- `sources.md`

### 2.2 服务层

回答：

- 单服务做了什么
- 入口、依赖、状态、消息、表、规则、场景是什么

### 2.3 规范层

回答：

- 哪些模式已经被多个服务反复证明
- 哪些仍只是单服务代码事实总结
- 哪些是团队建议规范，还不是代码事实

规范文档必须标注类型：

- `code-fact-summary`
- `team-proposed-standard`

## 3. 强事实与线索级

必须区分：

- `强事实承载服务`
- `线索级关联服务`

不能因为存在依赖就自动归域。

## 4. 域页的最小结构

1. 域定义
2. 域内主服务
3. 域内能力归纳
4. 与其他域的关系
5. 当前边界
6. 使用建议
7. 来源追溯
